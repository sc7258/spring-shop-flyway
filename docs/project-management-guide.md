# Project Management Guide (Document-Driven)

이 문서는 복잡한 이슈 트래킹 도구 대신, **Markdown 문서 4종**을 활용하여 프로젝트를 효율적으로 관리하는 방법을 설명합니다.
1인 개발이나 소규모 팀에서 **속도와 유연성**을 확보하는 데 최적화되어 있습니다.

## 1. 문서 체계 (The 4 Documents)

| 문서 | 파일명 | 역할 | 비유 |
|---|---|---|---|
| **Roadmap** | `.ai/30-roadmap.md` | 장기적인 목표와 마일스톤 (Phase) | **지도 (Map)** |
| **Plan** | `.ai/31-plan.md` | 현재 Phase의 상세 실행 계획 및 순서 | **작전명령서** |
| **Todo** | `.ai/32-todo.md` | 당장 처리해야 할 세부 작업 목록 & 메모 | **메모장 (Notepad)** |
| **Changelog** | `.ai/33-changelog.md` | 완료된 작업과 변경 사항 기록 | **항해일지 (Log)** |

---

## 2. 워크플로우 (Workflow)

### Step 1: 큰 그림 그리기 (Roadmap)
- 프로젝트 시작 시 또는 Phase 완료 시 작성합니다.
- 전체 목표를 Phase 단위로 나누고, 각 Phase의 핵심 목표를 정의합니다.
- **예시:** "Phase 1에서는 회원가입과 로그인을 끝내자."

### Step 2: 상세 계획 수립 (Plan)
- 새로운 Phase가 시작될 때 작성합니다.
- Roadmap의 목표를 달성하기 위해 필요한 기술적 작업(DB설정, API구현 등)을 순서대로 나열합니다.
- **예시:** "1. DB 테이블 생성 -> 2. Entity 작성 -> 3. API 구현"

### Step 3: 실행 및 체크 (Todo) - *가장 빈번하게 사용*
- 매일 작업을 시작할 때 `31-plan.md`를 보고 할 일을 가져옵니다.
- **작업 중 발생하는 모든 것**을 여기에 적습니다.
  - 세부 할 일 체크리스트 (`- [ ]`)
  - 구현 중 발생한 에러 로그 및 해결 방법 메모
  - 테스트 시나리오 및 결과
- 작업이 완료되면 체크박스를 채우고(`- [x]`), 불필요한 메모는 정리합니다.

### Step 4: 기록 및 회고 (Changelog)
- 의미 있는 기능 단위가 완성되거나 버전을 릴리즈할 때 작성합니다.
- `32-todo.md`에서 완료된 항목들을 요약하여 옮겨 적습니다.
- **예시:** "v0.1.0: 회원가입 API 구현 및 JWT 인증 적용 완료."

## 3. 파일별 관리 규칙

### `.ai/30-roadmap.md`
- **수정 빈도:** 낮음 (Phase 전환 시)
- **내용:** 거시적인 관점 유지. 세부 구현 내용은 적지 않음.

### `.ai/31-plan.md`
- **수정 빈도:** 중간 (작업 순서 변경 시)
- **내용:** 현재 Phase에 집중. Phase가 넘어가면 내용을 덮어쓰기(Overwrite)하여 최신화.

### `.ai/32-todo.md`
- **수정 빈도:** 매우 높음 (실시간)
- **내용:** 자유로운 형식. 스크래치패드(Scratchpad)처럼 사용.
- **Tip:** 작업이 끝나면 내용을 비우거나, `## Done` 섹션으로 내려서 깔끔하게 유지.

### `.ai/33-changelog.md`
- **수정 빈도:** 기능 완료 시
- **내용:** 과거형으로 작성. ("~함", "~추가됨")
- **형식:** [Keep a Changelog](https://keepachangelog.com/ko/) 형식을 권장.

---

## 4. 장점
- **Context Switching 최소화:** IDE를 벗어나지 않고 문서와 코드를 오가며 작업 가능.
- **AI 협업 최적화:** AI에게 현재 상태(`todo.md`)와 계획(`plan.md`)을 보여주면 문맥을 완벽하게 이해함.
